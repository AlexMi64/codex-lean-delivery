# Lean Delivery

`lean-delivery` — это skill для Codex, который делает рабочие ответы короче, не ухудшая инженерное качество.

Он нужен для задач, где важен результат, скорость и экономия токенов, а не длинная narration по каждому шагу.

## Что это дает

Skill подталкивает Codex к тому, чтобы:

- писать короткие промежуточные апдейты
- не пересказывать запрос пользователя
- не вставлять логи и вывод команд целиком без причины
- давать компактный итог вместо длинного changelog
- сохранять явную проверку, риски и точные технические детали

Главный принцип: **сжимать ответ, а не качество**.

## Для кого этот skill

`lean-delivery` полезен, если вы часто используете Codex для:

- реализации фич
- исправления багов
- рефакторинга
- отладки
- code review
- подготовки изменений к commit / push / deploy

Если вам не нужен длинный tutorial-style ответ, этот skill обычно дает более плотную и удобную подачу.

## Когда использовать

Используйте `lean-delivery`, когда хотите:

- меньше расходовать токены
- быстрее проходить рабочие итерации
- сократить болтовню в процессе
- получать короткий, но внятный итог по результату работы

Лучший способ использовать skill:

- вызывать его явно через `$lean-delivery`
- включать перед конкретной задачей, где важна краткость

## Когда не использовать

Skill не подходит для задач, где пользователь хочет:

- подробное обучение
- разбор "по шагам"
- ELI5-объяснение
- длинное сравнение подходов
- архитектурное исследование
- RFC, design doc или стратегическую записку

Если задача требует глубины, skill должен перестать агрессивно сжимать ответ.

## Что нового в v2

Во второй версии skill стал заметно безопаснее для публичного использования:

- сужены условия срабатывания
- implicit invocation отключен по умолчанию
- добавлены режимы `light`, `lean` и `risk-aware`
- добавлена явная `token strategy`
- усилены ограничения для code review
- добавлены примеры хорошего и плохого сжатия

## Как это работает

Skill экономит токены в таком порядке:

1. Убирает повторы.
2. Схлопывает рутинное повествование.
3. Резюмирует вывод команд и логов.
4. Только после этого дополнительно сжимает формулировки.

Skill **не** должен экономить токены за счет:

- пропуска проверки
- сокрытия неопределенности
- потери важных рисков
- расплывчатых формулировок
- слишком короткого ответа, который вызывает лишний follow-up

## Режимы сжатия

### `light`

Легкое сжатие. Подходит, когда ответ должен быть короче обычного, но задаче все еще нужен небольшой контекст.

### `lean`

Основной режим. Подходит для реализации, дебага и быстрых рабочих итераций.

### `risk-aware`

Краткий, но осторожный режим. Используется в рискованных задачах, где нельзя скрыть проверку, влияние изменения или непроверенные допущения.

## Поведение в рискованных задачах

Skill должен становиться менее агрессивным, если задача касается:

- аутентификации и авторизации
- платежей и биллинга
- разрушительных изменений данных
- миграций
- деплоя и инфраструктуры
- секретов и production-конфигов
- чувствительных участков безопасности

В таких случаях краткость допустима, но она не должна скрывать:

- что именно изменилось
- что было проверено
- что не было проверено
- какой риск остался

## Поведение в code review

Для code review skill должен сжимать только подачу, но не сам анализ.

Это означает:

- нельзя уменьшать количество находок
- нельзя убирать severity
- нельзя делать замечания расплывчатыми

Правильный эффект в review:

- меньше воды
- та же глубина
- более плотные и полезные формулировки

## Пример ожидаемого эффекта

Вместо такого ответа:

> Я исследовал проблему и обнаружил, что ключ кеша не включал tenant ID, из-за чего могли возникать коллизии между арендаторами. Я обновил логику кеширования, чтобы ключ включал tenant и resource ID, и затем прогнал целевой unit test.

Skill должен стремиться к такому:

> Исправил ключ кеша: теперь он включает tenant ID и resource ID, поэтому межтенантные коллизии больше не возникают. Проверено целевым unit test.

Больше примеров: [references/examples.md](references/examples.md)

## Установка

Скопируйте папку skill в директорию пользовательских skills Codex:

```bash

cp -R lean-delivery ~/.agents/skills/
```

Если skill не появился сразу, перезапустите Codex.

## Ограничения

`lean-delivery` не является универсальным style override для всех задач.

Он не должен:

- ломать обучение и объяснения
- делать ответы телеграфными любой ценой
- скрывать важную неопределенность
- заменять отдельные skills для explanation, review или release safety

Лучше всего он работает как узкий рабочий skill, который включается тогда, когда действительно нужна краткость.
